## 6. AI 协作复盘

### AI 在哪些阶段帮助最大

1. **页面骨架批量生成（Day2）：** 一次性生成 8 个页面 + 3 个布局组件 + 完整路由配置，效率远高于手动逐个创建
2. **Mock 数据设计（Day3）：** 快速生成结构完整、字段合理的 JSON 数据，覆盖四类业务场景
3. **表单设计与校验（Day4）：** 四类业务的动态表单切换逻辑和完整校验规则
4. **状态管理架构（Day5）：** Store 设计思路清晰，边界划分合理
5. **交互组件（Day6）：** LoadingState、ErrorState、SearchBar 组件结构规范
6. **文档整理（Day7）：** README 重写、证据归档、检测报告生成

### AI 生成内容中出现过的问题

1. **业务数据不合规：** Day3 生成的跑腿数据中包含"代写作业"任务，不符合校园规范，人工替换为"代打印文件"
2. **字段命名与已有数据不一致：** Day4 任务说明示例字段名（from/to）与 db.json 已有字段名（pickupLocation/deliveryLocation）不同，AI 跟随了示例而非已有数据，需要人工统一
3. **过度设计倾向：** AI 多次生成 JWT Token、权限路由守卫、密码加密等代码，超出实训范围，需要人工限制
4. **未使用的变量残留：** Day7 构建检查发现 AppHeader 中未使用的 import 和 GroupBuyView 中未使用的变量
5. **README 功能描述可能夸大：** AI 在生成 README 时会自动补充"支付、实时聊天"等功能，需要逐项核对删除

### 自己如何判断和修改 AI 生成内容

1. **先明确边界再让 AI 生成：** 每次协作前先告诉 AI"这个阶段做什么、不做什么"
2. **逐文件 diff 检查：** 批量生成后用 git diff 逐个文件审查修改内容
3. **立即构建验证：** 每次代码生成后运行 `vue-tsc` 和 `vite build`，发现类型错误立即修复
4. **模拟用户操作：** 完成一个功能后模拟完整的用户操作流程，检查业务闭环
5. **数据内容合规审查：** 逐条检查 Mock 数据是否适合校园场景

### 哪些内容必须由自己理解后才能完成

1. 校园业务场景的判断——什么内容是合适的、什么是不合适的
2. 数据字段的命名规范——需要理解已有 db.json 的结构
3. Pinia Store 的边界划分——什么数据放 Store、什么数据放组件内部
4. 功能边界的确定——每个 Day 阶段的目标和范围
5. README 和证据卡的真实性——不能写未实现的功能

### 以后使用 AI Coding 工具时应注意什么

1. **先规划再执行：** 不能让 AI"自由发挥"，每次协作前人工明确任务边界
2. **分批验证、不一次性大改：** 每次改完一批文件就构建验证，而不是等到最后
3. **保持项目始终可运行：** 不为了追新功能让项目崩溃
4. **AI 是效率工具，不是决策工具：** 代码的正确性、合规性、可维护性由人负责
5. **证据和文档必须真实：** AI 生成的文档需要人工逐项核对，不能虚构

## 8. 项目不足与改进方向

### 技术层面

| 不足 | 说明 | 改进方向 |
|------|------|----------|
| Mock 数据 | 使用 JSON Server，不是真实后端 | 接入真实 RESTful API 或 BaaS 服务 |
| 用户认证 | 明文密码 + localStorage，非安全方案 | 引入 JWT Token + 密码加密 |
| 收藏持久化 | 刷新后收藏丢失（仅 Pinia 内存） | localStorage 持久化或后端存储 |
| 图片上传 | 使用占位图地址 | 实现图片上传功能 |
| 搜索功能 | 仅前端过滤，不支持模糊搜索 | 后端全文检索或更完善的前端搜索 |
| 消息中心 | 静态示例数据 | 接入实时消息推送 |
| 商品详情 | 占位内容，未对接 API | 从 API 获取完整商品详情 |

### 工程层面

| 不足 | 说明 | 改进方向 |
|------|------|----------|
| 测试覆盖 | 无自动化测试 | 添加单元测试和 E2E 测试 |
| 响应式适配 | 固定 480px max-width | 适配更多屏幕尺寸 |
| 错误处理 | 仅二手交易页有完整状态处理 | 其他页面也添加 loading/error/empty 状态 |
| 组件文档 | 组件缺少使用说明 | 添加组件级文档或 Storybook |

## 9. 今日反思

通过 7 天实训，我对前端项目开发、AI 协作开发和工程化过程有了更深入的理解：

**关于前端工程实践：** 一个前端项目的完成不仅仅是"页面能打开"，它还包括路由设计是否合理、组件是否可复用、状态管理边界是否清晰、数据流是否一致、错误状态是否覆盖、构建是否通过、文档是否完整。Day7 的综合验收让我意识到，项目的"完整性"比"功能多少"更重要。

**关于 AI 协作开发：** AI 在批量代码生成、结构设计、文档整理方面效率极高，但它不理解业务上下文、不会判断数据合规性、不知道功能边界。7 天的实践让我学会了"先规划再让 AI 执行、分批验证不一次性大改、人工审查每一处修改"的协作模式。AI 是效率工具，但代码质量、业务正确性和项目完整性的最终责任在开发者。

**关于工程规范：** Git 提交记录不是形式要求，而是开发过程的真实证据。证据卡不是为了应付检查，而是帮助自己梳理"做了什么、为什么这样做、遇到了什么问题"。README 写清楚项目功能和运行方式，既是给别人看的，也是给自己梳理的。

**关于项目思维：** 从 Day1 的理解项目到 Day7 的综合验收，我经历的不仅是一个前端开发过程，更是一个完整的工程实践闭环——理解需求 → 规划架构 → 逐步实现 → 持续验证 → 归档整理 → 展示复盘。这个闭环在任何软件工程项目中都适用。
