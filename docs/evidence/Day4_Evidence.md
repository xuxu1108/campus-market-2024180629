## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | trades | title、category、price、condition、location、description | 二手商品需要展示物品分类、价格和成色，便于买家判断物品价值 |
| 失物招领 | lostFounds | title、type、itemName、location、time、contact、description | 需区分"寻物"和"招领"两种场景，物品名称和时间是关键查找线索 |
| 拼单搭子 | groupBuys | title、type、targetCount、deadline、location、description | 拼单需要明确参与人数目标和截止时间，类型标识拼单/搭子/组队 |
| 跑腿委托 | errands | title、taskType、reward、pickupLocation、deliveryLocation、deadline、description | 跑腿涉及取件和送达两个地点，酬劳是接单者的直接动力 |

### 字段设计思路

发布表单不是简单的"把所有字段堆在一起"。核心设计原则是：

1. **通用字段统一**：标题、地点、描述是所有发布类型都需要的，放在表单前部供所有类型共用
2. **专属字段按需展示**：根据用户选择的发布类型，使用 `v-if` 条件渲染对应字段，避免无关字段干扰
3. **字段命名与 db.json 一致**：例如跑腿使用 `pickupLocation`/`deliveryLocation`（而非 `from`/`to`），失物招领使用 `time`（而非 `eventTime`），确保提交数据与已有数据结构一致
4. **发布人固定值**：Day4 阶段使用"当前用户"作为固定值，为后续 Day5 的 Pinia 用户状态管理预留接口

## 3. 表单校验规则

校验逻辑遵循"必要的才校验，不增加无用规则"的原则：

### 通用校验（所有类型适用）
- **标题不能为空**：列表页依赖标题展示，空标题会导致列表项无意义
- **地点不能为空**：校园交易依赖地理位置信息撮合用户
- **描述不能为空**：描述是用户判断信息可信度的关键依据

### 二手交易专属校验
- **商品分类不能为空**：分类用于列表页筛选，空分类影响检索
- **价格必须大于 0**：价格是二手交易的核心属性，≤0 无意义
- **成色不能为空**：成色直接影响买家决策

### 失物招领专属校验
- **物品名称不能为空**：失物招领靠物品名称匹配
- **发生时间不能为空**：时间信息帮助判断物品丢失/发现的时效性

### 拼单搭子专属校验
- **拼单类型不能为空**：用于区分拼单/搭子/组队场景
- **目标人数不能少于 2**：少于 2 人失去"拼"的意义
- **截止时间不能为空**：拼单活动有时效性

### 跑腿委托专属校验
- **任务类型不能为空**：区分取快递/代买/代送等场景
- **酬劳不能为负数**：酬劳不应为负数
- **取件地点不能为空**：跑腿需要知道从哪里取
- **送达地点不能为空**：跑腿需要知道送到哪里
- **截止时间不能为空**：任务有时效性

## 7. 遇到的问题与解决方法

### 问题 1：字段名不一致导致 db.json 数据不统一
**现象**：初始设计时，任务说明给出的示例字段名（from/to/eventTime）与项目中 db.json 已有的字段名（pickupLocation/deliveryLocation/time）不一致。

**原因**：Day4 任务说明中的示例代码使用了简化字段名，但项目 db.json 在 Day3 时已建立了自己的字段命名规范。

**解决方法**：以 db.json 中已有字段名为准，修改了表单绑定、校验逻辑和 POST 提交中的字段名，确保与已有数据结构完全一致。

### 问题 2：发布类型切换后表单数据残留
**现象**：先选择"二手交易"填写了价格和成色，再切换到"失物招领"，虽然专属字段不显示，但数据仍然存在于 form 对象中。

**解决方法**：使用 Vue 的 `watch` API 监听 `publishType` 变化，自动调用 `resetForm()` 清空所有字段。

## 8. 今日反思

发布表单、表单校验和数据新增是 Web 应用中承上启下的关键环节。在此之前，项目只能"看"数据（从 JSON Server 读取并展示列表），但无法"写"数据（让用户产生新内容）。Day4 的核心价值在于打通了"用户输入 → 前端校验 → 接口提交 → 数据持久化 → 列表展示"的完整闭环。

发布表单的设计不只是堆几个输入框。它需要考虑业务类型差异（四类信息的字段不同）、用户体验（动态切换展示对应字段）、数据一致性（字段名与后端/JSON Server 一致）等多个维度。表单校验也不只是简单的"必填判断"——每个校验规则背后都有业务含义，例如"拼单人数不能少于 2"是因为少于 2 人的拼单失去意义，"价格必须大于 0"是因为交易需要明确的价值交换。

数据新增是将用户输入转化为结构化数据的关键一步。通过 Axios 向 JSON Server 发送 POST 请求，数据被写入 db.json 并持久化保存。这一步验证了前端表单的字段设计是否正确——如果字段名与 db.json 不一致，列表页就无法正确展示新增数据。

Day4 的实现为后续奠定了数据基础，Day4 的核心不是追求字段多少，而是确保"发布流程可用、字段贴合业务、校验逻辑清晰、数据新增可验证"。
